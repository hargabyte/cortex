Cortex is a codebase intelligence tool that helps development teams and AI agents understand complex software projects. It analyzes source code, builds a knowledge graph of how components relate to each other, and provides insights about code structure, dependencies, and potential risks.

The system is primarily written in Go (95% of entities), with supporting components in Python (4%) and TypeScript (1%). This language distribution reflects a deliberate architectural choice: Go provides the performance and reliability needed for core analysis operations, while Python and TypeScript handle specialized parsing tasks.

Understanding the Diagram

The architecture follows a layered design where data flows from user commands through processing layers to the underlying data store. Let me walk you through each major component:

Key Insight: The system is built around a central data store (the "Store" component) that acts as the single source of truth. All other modules interact with this store to read or write information about the codebase.

Module-by-Module Overview

cmd (Command Layer)

The entry point where users interact with Cortex. Commands like find, show, safe, and report all start here. The Execute function routes user requests to the appropriate handler.

parser (Code Analysis)

Transforms source code files into structured data. The walkNode function (the most connected component in the system with 118 dependencies) traverses code syntax trees to extract meaningful information.

extract (Entity Extraction)

Identifies functions, types, and relationships from parsed code. The nodeText method (502 connections) is the workhorse that extracts text from code elements.

store (Data Persistence)

The central database that stores all codebase knowledge. The Store type (171 connections) and Entity type (280 connections) are the foundation that everything else builds upon.

output (Formatting)

Handles how information is displayed to users. The Density type controls how much detail is shown - from sparse summaries to dense technical output.

daemon (Performance)

An optional background service that keeps the database connection warm, enabling faster response times for frequently-used commands.

Data Flow Through the System

When a user runs a command like cx find LoginUser:

  1. cmd receives the command and parses arguments
  2. store is queried to find matching entities
  3. graph calculates importance and relationships
  4. output formats results based on the requested density
  5. Results are displayed to the user

For code scanning (cx scan):

  1. parser reads source files and builds syntax trees
  2. extract identifies entities (functions, types, etc.)
  3. store persists the extracted information
  4. graph computes relationships and importance metrics
